# Organization Scorecard

This scorecard keeps the MT5 company lean and useful instead of merely large.

## What Good Looks Like

- Shared skills are distinct and easy to route.
- Knowledge from one EA family is reused by the next one.
- Validation quality improves faster than org complexity.
- The company can explain why a role, MCP, or workflow still exists.

## Core Metrics

- `roster clarity`
  Every shared skill should be in one of three states: `core`, `watch`, or `candidate`.
- `knowledge reuse`
  Every serious EA candidate should cite at least one reusable lesson from `knowledge/patterns/` or `knowledge/company/`.
- `validation coverage`
  Each promoted candidate should have a visible state for search, locked validation, OOS, and forward/demo.
- `tool reliability`
  Compile, backtest, optimization, import, and telemetry workflows should be reproducible from repo scripts.
- `queue discipline`
  Active items in `.company/secretary/queue.md` should represent real unresolved work, not stale ideas.
- `org drag`
  If a skill or MCP is not changing decisions, it should be merged, downgraded to watch status, or removed.

## Add / Keep / Remove Rules

- Add a new shared skill only when one of these is true:
  - the same missing expertise appears in multiple reviews
  - routing repeatedly lands on the wrong existing skill
  - a live-readiness gate cannot be owned cleanly by the current roster
- Keep a skill as `core` when it improves decisions repeatedly and its scope is clear.
- Move a skill to `watch` when it overlaps with another skill or is rarely the best owner.
- Remove or merge a skill when it creates naming noise without changing output quality.
- Add a new shared MCP only when it removes recurring manual work or unlocks missing evidence.

## Review Triggers

- after any shared skill add/remove/merge
- after any shared MCP add/remove/replacement
- before live-promotion decisions
- at major strategy family pivots
- at least once per quarter, even if no org change was made

## Output Requirement

Each review should leave:

- a snapshot or review note under `.company/improvement/`
- a reusable summary under `knowledge/company/` or `knowledge/patterns/`
- a CEO approval entry when root policy changed
